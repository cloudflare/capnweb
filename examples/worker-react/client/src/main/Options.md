# Options

## Change the Import

If a user has

```typescript
import { newHttpBatchRpcSession } from 'capnweb'
```

and they want to add validation.  They can do it by changing the import.

This is nice because it's a single line of change for all invocations.

The generator runs on the server code (can be as a build tool plugin).

### Option A: Separate Package

```typescript
import { newHttpBatchRpcSession } from 'capnweb-validate'
```

- pros
  - separate package, separate dependencies (unpkg)
  - in the future, supporting ts-go may require more dependencies (on tsgo)
- cons
  - need to deploy another package
  - needs to keep in sync with capnweb (as a peer dependency)

### Option B: ESM path

```typescript
import { newHttpBatchRpcSession } from 'capnweb/validate'
```

- pros
  - same package
  - no need for synced peer dependency
- cons
  - adds one dependency to `capnweb` (unpkg)


### In both cases

We can override with a method decorator:


```typescript
import { skipRpcValidation } from '{option a or optoin b}';

export class Api extends RpcTarget {
  @skipRpcValidation()
  async authenticate(sessionToken: string): Promise<User> {
    // ...
  }
```

There probably isn't a need for a class level validation skip because they can just decide that by where they import from.

## Add As Decorator

If a user has

```typescript
export class Api extends RpcTarget {
  async authenticate(sessionToken: string): Promise<User> {
    // ...
  }
```

and they want to add validation, they can do it by one of the following approaches

### Option A: Class Validation

```typescript
import { Validate } from 'capnweb-validate';

@Validate()
export class Api extends RpcTarget {
  async authenticate(sessionToken: string): Promise<User> {
    // ...
  }
```

- pros
  - very straightforward
- cons
  - requires the user to change their source code

### Option B: Method Validation

```typescript
import { Validate } from 'capnweb-validate';

export class Api extends RpcTarget {
  @Validate()
  async authenticate(sessionToken: string): Promise<User> {
    // ...
  }
```

- pros
  - finer-grained control
- cons
  - requires the user to change their source code

### Option C: Both

We can also allow the same decorator to be used on either methods or classes.

## Generate the Validator, then the User Inserts

If we want to make it crystal clear what's happening, we can generate a file like this

```typescript
// validate.ts
// THIS FILE IS GENERATED BY CAPNWEB. DO NOT EDIT THIS FILE DIRECTLY.

import { v } from "capnweb-validate/internal";

export const validate = {
  serviceName: "Api",
  methods: {
    "authenticate": {
        args: [v.string],
        returns: v.object({ "id": v.string, "name": v.string }, "User")
    },
    "getUserProfile": {
        args: [v.string],
        returns: v.object({ "id": v.string, "bio": v.string }, "Profile")
    },
    "getNotifications": {
        args: [v.string],
        returns: v.array(v.string)
    },
  },
};
```

and users would ad the validation as a second argument parameter to Api constructor:

```typescript
import { validate } from './validate.ts';

const resp = await newWorkersRpcResponse(
    request,
    new Api(env, validate)
);
```

or we could add it as a third parameter to the response generator:

```typescript
const resp = await newWorkersRpcResponse(
    request,
    new Api(env),
    validate
);
```

or perhaps we move to using an object bag in the constructor to make it more future-proof:

```typescript
import { validate } from './validate.ts';

const resp = await newWorkersRpcResponse(
    request,
    new Api({
        env,
        validate,
    })
);
```
